# Part 1 — 기초 문법: 말이 되는 문장 만들기

> 문법 책 다 읽고 말 못하는 사람 많다. 여기선 딱 쓸 것만 한다.

---

## Chapter 1. 시제 — 딱 3개만 잡아라

시제가 6개라고? 실무 대화에서 **80%는 이 3개**로 다 된다. 나머지는 나중에 자연스럽게 따라온다.

---

### ⚡ 핵심 3개 먼저

**① 지금 하고 있어 → `-ing`**

> "I'm working on the onboarding flow right now."
> "We're still waiting for feedback from the PM."

**② 이미 했어 → `-ed`**

> "I finished the wireframes yesterday."
> "We decided to go with the simpler version."

**③ 앞으로 할 거야 → `will`**

> "I'll send you the file after the call."
> "I'll take care of that."

---

💡 **이것만 기억해:** 말하기 전에 **상황(지금/어제/나중)** 을 먼저 생각해라. 그 다음 패턴을 고른다. 한국어로 번역하려다 막히는 거다.

---

### 📌 실무에서 바로 쓰는 시제별 표현

**지금 → `-ing`**

| 상황 | 표현 |
|------|------|
| 작업 현황 공유 | I'm **working on** the new dashboard. |
| 진행 중인 변경 | We're **updating** the design system this week. |
| 기다리는 중 | I'm still **waiting for** dev feedback. |
| 리뷰 중 | I'm **looking at** the version you shared. |

**어제/지난번 → `-ed`**

| 상황 | 표현 |
|------|------|
| 완료 보고 | I **finished** the wireframes. |
| 이전 결정 | We **decided** to go with option B. |
| 테스트 결과 | Users **struggled** to find the CTA. |
| 피드백 전달 | The PM **mentioned** it was confusing. |

**나중/곧 → `will` 또는 `be going to`**

| | will | be going to |
|---|------|------------|
| 언제 | 즉흥적 결정, 그 자리에서 약속 | 이미 계획된 것 |
| 예문 | I **will** send you the file now. | I**'m going to** present on Friday. |
| 느낌 | "그럼 내가 할게" | "원래 할 예정이야" |

---

### ⚠️ 이 실수 많이 해

```
❌ I am finish the design.
✅ I finished the design.

❌ I working on it right now.
✅ I'm working on it right now.

❌ Yesterday I am in a meeting.
✅ Yesterday I was in a meeting.
```

---

### 🎯 보너스: 현재 완료 (have + p.p.)

"이미 했는데, 그게 지금도 영향을 줄 때" 쓴다.

> "I **have just uploaded** the new screens to Figma." → 방금 올렸으니까 지금 봐도 돼
> "**Have you reviewed** the latest prototype?" → 아직 봤어? (지금 확인 요청)
> "I **haven't received** any feedback yet." → 아직 못 받았어 (지금도 기다리는 중)

💡 **단순 과거 vs 현재 완료 구분:**
- "I **finished** the design." → 언제 끝냈는지가 중요
- "I **have finished** the design." → 지금 준비됐다는 상태가 중요

---

### ✏️ 5초 드릴 — 한국어 보고 바로 영어로

1. 지금 온보딩 플로우 작업 중이야. → `I'm working on the onboarding flow.`
2. 어제 와이어프레임 다 끝냈어. → `I finished the wireframes yesterday.`
3. 파일 지금 바로 보낼게. → `I'll send you the file right now.`
4. 방금 피그마에 올렸어. → `I've just uploaded it to Figma.`
5. 아직 피드백 못 받았어. → `I haven't received any feedback yet.`

## Chapter 2. 조동사 — 이것 하나로 말의 온도를 조절한다

영어 잘하는 사람과 못하는 사람의 차이? **조동사** 다. 같은 말도 조동사 하나로 차갑게도, 따뜻하게도, 프로답게도 만들 수 있다.

---

### ⚡ 핵심 정리 — 온도계로 외워라

```
must      → 반드시 해야 해 (강제) 🌡️🌡️🌡️🌡️🌡️
should    → ~하는 게 좋아 (권고) 🌡️🌡️🌡️
could     → ~할 수도 있어 (부드러운 제안) 🌡️🌡️
might     → ~일 수도 있어 (조심스러운 가능성) 🌡️
would     → ~하겠어 / ~하면 어떨까 (정중) 🌡️🌡️
```

---

### 📌 피드백 줄 때 — 조동사로 온도 조절하기

같은 말, 완전히 다른 느낌:

| 표현 | 느낌 |
|------|------|
| You **must** change this button color. | 명령. 관계 상할 수 있음 ❌ |
| You **should** change this button color. | 권고. 일반적 피드백 ✅ |
| We **could** try a different color. | 제안. 팀 분위기 좋음 ✅ |
| This **might** work better in a different color. | 가장 조심스러운 의견 ✅ |

---

### 📌 상황별 바로 쓰는 표현

**should — 제안, 개선점 말할 때**

> "This **should** be more prominent."
> "The error message **should** be clearer."
> "We **should** test this before finalizing."
> "This **should** be consistent with the rest of the layout."

**could — 아이디어 던질 때**

> "We **could** try a different color for the CTA."
> "This **could** work better as a bottom sheet."
> "**Could** you share the latest version in Figma?"
> "We **could** schedule a quick sync to align on this."

**would — 정중하게 요청할 때**

> "**Would** you be able to review this by Friday?"
> "I **would** go with option B personally."
> "I **would** suggest simplifying the navigation."
> "That **would** be really helpful, thank you."

**might — 확신 없이 의견 낼 때**

> "This **might** confuse first-time users."
> "We **might** want to reconsider this approach."
> "I **might** be able to join the call, but I'm not sure."

---

### ⚠️ 실수 주의

```
❌ "You must fix this." (팀원한테 이렇게 말하면 관계 어색해짐)
✅ "This should be fixed before launch."

❌ "Can you do this?" (너무 직접적)
✅ "Could you take a look at this?" (훨씬 자연스러움)
```

---

### ✏️ 5초 드릴 — 온도에 맞는 조동사 고르기

1. 팀원한테 버튼 크기 키우라고 부드럽게 말하고 싶어. → `The button could be a bit bigger.`
2. 론칭 전에 테스트해야 한다고 권고하고 싶어. → `We should test this before launch.`
3. 금요일까지 검토해줄 수 있냐고 정중하게 요청하고 싶어. → `Would you be able to review this by Friday?`
4. 유저가 헷갈릴 수도 있다고 조심스럽게 말하고 싶어. → `This might confuse users.`

## Chapter 3. 질문하는 법 — 모르면 물어라, 단 프로답게

**"잘 못 알아들었는데 어떻게 하지?"** 이 걱정 때문에 회의에서 가만히 있는 사람 많다. 근데 모르는 척 넘어가면 더 큰 문제가 생긴다. 물어보는 게 훨씬 프로답다.

---

### ⚡ 질문 유형 4가지만 기억해

| 유형 | 언제 | 패턴 |
|------|------|------|
| 확인 | Yes/No 알고 싶을 때 | Did you / Is this / Are we... |
| 정보 | 뭔가를 알고 싶을 때 | What / Who / When / Why / How... |
| 정중한 질문 | 격식 있는 자리 | Could you tell me / I was wondering if... |
| 내 이해 확인 | 잘 이해했는지 체크 | So what you're saying is...? |

---

### 📌 상황별 바로 쓰는 질문

**이해가 안 됐을 때 — 이게 제일 중요**

> "Could you clarify what you mean by that?"
> "Just to make sure I understand — are you saying ~?"
> "Could you give me an example?"
> "I want to make sure I'm on the same page."

**못 들었을 때 — 당황하지 마라**

> "Sorry, could you repeat that?"
> "I missed the last part — could you say that again?"
> "Sorry, the connection cut out — what was that?" *(화상 회의에서)*
> "Could you slow down a bit? I want to make sure I catch everything."

**결정/우선순위 확인할 때**

> "What's the priority here?"
> "Who's the decision maker on this?"
> "Are we aligned on this?"
> "Is this set in stone, or is there still room to explore?"

**일정 물을 때**

> "What's the timeline for this?"
> "When do you need this by?"
> "Is there any flexibility on the deadline?"

---

### 💡 간접 질문 — 이렇게만 바꿔도 훨씬 프로답게 들린다

직접 물으면 OK. 간접으로 물으면 Better.

| 직접 (OK) | 간접 (Better) |
|-----------|--------------|
| What is the deadline? | Could you tell me what the deadline is? |
| Why did you change this? | I was wondering why this was changed. |
| Who approved this? | Do you know who approved this? |
| What does this mean? | Could you help me understand what this means? |

---

### ⚠️ 절대 하면 안 되는 것

```
❌ 모르는데 그냥 "Yes, yes" 하기 → 나중에 더 큰 문제
❌ "I don't understand" 하고 끝내기 → 구체적으로 뭘 모르는지 말해야 함
✅ "I understand the overall direction, but could you clarify the last part?"
```

---

### ✏️ 5초 드릴

1. PM이 한 말이 모호해. 뭘 의미하는지 물어봐. → `Could you clarify what you mean by that?`
2. 회의에서 데드라인을 물어봐야 해. → `When do you need this by?`
3. 내 이해가 맞는지 확인하고 싶어. → `Just to make sure I understand — are you saying we're going with option B?`
4. 화상 회의에서 끊겼어. → `Sorry, the connection cut out — could you repeat that?`

## Chapter 4. 수동태 — 문서에 이게 가득하다, 읽을 줄 알아야 한다

피그마 핸드오프 문서, 지라 티켓, 스펙 시트 — 다 수동태 투성이다. 내가 쓸 줄 알면 더 좋고, 적어도 읽을 줄은 알아야 한다.

---

### ⚡ 수동태 쓰는 이유 딱 2가지

**① 누가 했는지보다 무엇이 됐는지가 중요할 때**
> "The design **was approved**." (누가 승인했는지보다 승인됐다는 사실이 중요)

**② 공식 문서, 책임 소재를 흐릴 때**
> "The deadline **was pushed**." (누가 밀었는지 굳이 안 밝힐 때)

---

### 📌 패턴 — 이것만 기억해

```
주어 + be동사 + 과거분사(p.p.)
```

| 시제 | 패턴 | 예문 |
|------|------|------|
| 현재 | is/are + p.p. | This feature **is triggered** by a tap. |
| 과거 | was/were + p.p. | The bug **was reported** during QA. |
| 완료 | has/have been + p.p. | The component **has been updated**. |
| 미래 | will be + p.p. | The feature **will be released** next Thursday. |
| 필요 | needs to be + p.p. | This **needs to be approved** by the PM. |

---

### 📌 실무에서 자주 보이는 수동태 문장

**리뷰/승인**
> "This design **has been reviewed** and approved."
> "Changes **were requested** by the stakeholders."
> "The design **needs to be approved** before handoff."

**개발 협업**
> "Specs **should be finalized** before development starts."
> "This component **is expected** to be reusable."
> "The API **has been integrated** into the prototype."

**일정/배포**
> "The release date **has been pushed** to next week."
> "The feature **was shipped** in the last sprint."
> "Updates **will be communicated** via Slack."

**이슈**
> "A bug **was reported** during testing."
> "A fix **has been deployed** to staging."
> "This edge case **hasn't been handled** yet."

---

### ⚠️ 능동태 vs 수동태 — 언제 뭘 쓸까

```
내가 한 일 말할 때 → 능동태
  ✅ "I updated the mockup."

누가 했는지 모르거나 중요하지 않을 때 → 수동태
  ✅ "The mockup was updated."

공식 문서, 스펙 → 수동태
  ✅ "Designs must be reviewed before handoff."
```

---

### ✏️ 5초 드릴 — 능동태를 수동태로

1. The designer updated the components. → `The components were updated.`
2. We will release the new feature next week. → `The new feature will be released next week.`
3. The team has approved the final design. → `The final design has been approved.`
4. Someone reported a bug. → `A bug was reported.`
