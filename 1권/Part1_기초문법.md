# 1권 Part 1 — 기초 문법: 말이 되는 문장 만들기

> 문법은 외우는 게 아니라 패턴을 익히는 것. 디자이너 실무 상황에 바로 꽂히는 예문으로 감각을 쌓는다.

---

## Chapter 1. 현재/과거/미래: 업무 상황별 시제 완전 정복

### 🔑 핵심 원칙

| 시제 | 언제 쓰나 | 한 줄 공식 |
|------|-----------|-----------|
| 단순 현재 | 사실, 반복, 프로세스 설명 | "이건 항상/보통 이래" |
| 현재 진행 | 지금 진행 중인 작업 | "지금 하고 있어" |
| 단순 과거 | 완료된 일, 이미 일어난 일 | "그때 그랬어" |
| 현재 완료 | 과거에 한 일이 지금도 영향을 줌 | "이미 했어 (그래서 지금 이래)" |
| will | 즉흥적 결정, 약속 | "그럼 내가 할게" |
| be going to | 이미 계획된 일 | "원래 ~할 예정이야" |

---

### 📌 단순 현재 (Simple Present)

> 사실, 기능 설명, 반복되는 업무 프로세스에 사용

**패턴:** `주어 + 동사(s)`

| 상황 | 표현 |
|------|------|
| 기능 설명 | This button **opens** the settings panel. |
| 프로세스 설명 | We **conduct** usability testing every sprint. |
| 의견 말하기 | I **think** this layout feels too crowded. |
| 사용자 행동 묘사 | Users **expect** to find the search bar at the top. |
| 팀 루틴 | Our team **syncs** every Monday at 10. |

---

### 📌 현재 진행 (Present Progressive)

> 지금 작업 중인 것, 현재 진행 중인 프로젝트

**패턴:** `주어 + am/is/are + 동사ing`

| 상황 | 표현 |
|------|------|
| 작업 현황 공유 | I'm **working on** the onboarding flow right now. |
| 진행 중인 변경 | We're **updating** the design system this week. |
| 리뷰 중 | I'm **looking at** the latest version you shared. |
| 기다리는 중 | We're still **waiting for** feedback from the PM. |
| 협업 중 | The dev team **is building** the component as we speak. |

---

### 📌 단순 과거 (Simple Past)

> 완료된 작업, 지난 회의, 이전 결정 사항 언급

**패턴:** `주어 + 동사ed / 불규칙 과거형`

| 상황 | 표현 |
|------|------|
| 작업 완료 보고 | I **finished** the wireframes yesterday. |
| 지난 결정 언급 | We **decided** to go with the simpler version. |
| 피드백 전달 | The PM **mentioned** that the flow was confusing. |
| 테스트 결과 | Users **struggled** to find the CTA button. |
| 이전 버전 설명 | The old design **had** too many steps. |

---

### 📌 현재 완료 (Present Perfect)

> 과거에 한 일이 지금 상황에 영향을 줄 때. 한국어엔 없는 개념이라 많이 헷갈리는 시제.

**패턴:** `주어 + have/has + 과거분사(p.p.)`

| 상황 | 표현 |
|------|------|
| 방금 완료 | I **have just uploaded** the new screens to Figma. |
| 경험 | I **have worked on** B2B SaaS products before. |
| 변경 사항 공유 | We **have updated** the color tokens in the design system. |
| 아직 안 됨 | I **haven't received** any feedback yet. |
| 확인 요청 | **Have you reviewed** the latest prototype? |

> **💡 현재 완료 vs 단순 과거 구분 팁**
> - "I **finished** the design." → 언제 끝냈는지가 중요할 때
> - "I **have finished** the design." → 지금 준비됐다는 상태가 중요할 때

---

### 📌 미래: will vs be going to

| | will | be going to |
|---|------|------------|
| 느낌 | 즉흥적 결정, 약속 | 이미 계획된 일 |
| 예문 | I **will** send you the file right now. | I **am going to** present the designs tomorrow. |
| 예문 | **I'll** take care of that. | We **are going to** run user testing next week. |

---

### ✏️ 실전 패턴 연습

아래 빈칸에 알맞은 시제를 넣어보세요.

1. I ________ (just / finish) the high-fidelity mockups. → `have just finished`
2. We ________ (conduct) weekly design reviews every Thursday. → `conduct`
3. She ________ (work on) the new dashboard when the meeting started. → `was working on`
4. The team ________ (decide) to simplify the navigation last sprint. → `decided`
5. I ________ (send) you the updated file right now. → `will send`

## Chapter 2. 조동사로 말 부드럽게 하기

> 같은 말도 조동사 하나로 딱딱해지거나 부드러워진다. 특히 피드백 줄 때, 제안할 때 핵심.

### 🔑 조동사 뉘앙스 비교표

| 조동사 | 뉘앙스 | 강도 |
|--------|--------|------|
| must | 반드시 해야 한다 (강제) | ●●●●● |
| should | ~하는 게 좋다 (권고) | ●●●○○ |
| could | ~할 수도 있다 (가능성/제안) | ●●○○○ |
| would | ~하겠다 / ~하면 어떨까 (정중) | ●●○○○ |
| might | ~일 수도 있다 (약한 가능성) | ●○○○○ |
| can | ~할 수 있다 (능력/허가) | ●●●○○ |

---

### 📌 should — 제안, 권고

> 피드백 줄 때, 개선점 말할 때 가장 많이 쓰는 조동사

| 상황 | 표현 |
|------|------|
| 디자인 개선 제안 | This **should** be more prominent. |
| 사용자 관점 | The error message **should** be clearer. |
| 프로세스 제안 | We **should** run a quick usability test before finalizing. |
| 우선순위 | This **should** be addressed before the next sprint. |
| 부드러운 지적 | The spacing here **should** be consistent with the rest of the layout. |

---

### 📌 could — 가능성, 부드러운 제안

> "이렇게 해도 되지 않을까요?" 느낌. should보다 훨씬 부드러움.

| 상황 | 표현 |
|------|------|
| 대안 제시 | We **could** try a different color for the CTA. |
| 아이디어 던지기 | This **could** work better as a bottom sheet. |
| 가능성 언급 | Users **could** get confused by this flow. |
| 요청 | **Could** you share the latest version in Figma? |
| 회의 제안 | We **could** schedule a quick sync to align on this. |

---

### 📌 would — 정중한 표현, 조건

> 이메일, 공식 요청, 조건부 상황에서 자주 사용

| 상황 | 표현 |
|------|------|
| 정중한 요청 | **Would** you be able to review this by Friday? |
| 조건부 제안 | I **would** go with option B if we have more time. |
| 의견 표현 | I **would** suggest simplifying the navigation. |
| 감사 표현 | That **would** be really helpful, thank you. |
| 가정 | If we had more data, we **would** make a better decision. |

---

### 📌 might — 약한 가능성, 조심스러운 말

> 확실하지 않을 때, 조심스럽게 의견 낼 때

| 상황 | 표현 |
|------|------|
| 불확실한 예측 | This **might** confuse first-time users. |
| 조심스러운 제안 | We **might** want to reconsider this approach. |
| 가능성 언급 | There **might** be a better way to handle this edge case. |
| 약속 못할 때 | I **might** be able to join the call, but I'm not sure yet. |

---

### 📌 실전 비교: 같은 말, 다른 온도

피드백 상황: "이 버튼 색을 바꿔야 해"

| 강도 | 표현 | 느낌 |
|------|------|------|
| 강함 | You **must** change this button color. | 지시, 명령 → 관계 상할 수 있음 |
| 보통 | You **should** change this button color. | 권고, 일반적 피드백 |
| 부드러움 | We **could** change this button color. | 제안, 팀 분위기 좋음 |
| 가장 부드러움 | This **might** work better in a different color. | 조심스러운 의견 제시 |

## Chapter 3. 질문하는 법 — 회의에서 자연스럽게 묻기

> 모르면 물어야 한다. 근데 어떻게 묻느냐가 프로와 아마추어를 가른다.

### 🔑 질문 유형 4가지

| 유형 | 언제 | 예문 |
|------|------|------|
| Yes/No 질문 | 확인할 때 | Did you finalize the scope? |
| Wh- 질문 | 정보 얻을 때 | What's the deadline for this? |
| 간접 질문 | 정중하게 물을 때 | Could you tell me what the deadline is? |
| 확인 질문 (Tag) | 내 이해 맞는지 확인 | We're launching next week, right? |

---

### 📌 회의에서 바로 쓰는 질문 패턴

**이해 확인**

| 표현 | 상황 |
|------|------|
| Could you clarify what you mean by that? | 말이 모호할 때 |
| Just to make sure I understand — are you saying ~? | 내 이해 맞는지 확인 |
| So what you're saying is ~, correct? | 요약 후 확인 |
| Could you give me an example? | 추상적인 말을 구체화할 때 |

**의견/결정 확인**

| 표현 | 상황 |
|------|------|
| What's the priority here? | 우선순위 물을 때 |
| Who's the decision maker on this? | 결정권자 확인 |
| Are we aligned on this? | 모두 동의하는지 확인 |
| Is this set in stone, or is there still room to explore? | 결정이 확정인지 물을 때 |

**일정/범위 질문**

| 표현 | 상황 |
|------|------|
| What's the timeline for this? | 일정 물을 때 |
| What's in scope for this sprint? | 범위 확인 |
| When do you need this by? | 데드라인 확인 |
| Is there any flexibility on the deadline? | 데드라인 조정 가능한지 |

**모를 때 / 놓쳤을 때**

| 표현 | 상황 |
|------|------|
| Sorry, could you repeat that? | 못 들었을 때 |
| I missed the last part — could you say that again? | 일부를 놓쳤을 때 |
| I'm not familiar with that term — what does it mean? | 모르는 용어 나왔을 때 |
| Could you slow down a bit? | 너무 빠를 때 |

---

### 📌 간접 질문 — 부드럽게 묻는 법

직접 질문보다 간접 질문이 훨씬 프로답고 정중하게 들린다.

| 직접 질문 (OK) | 간접 질문 (Better) |
|---------------|-------------------|
| What is the deadline? | Could you tell me what the deadline is? |
| Why did you change this? | I was wondering why this was changed. |
| Who approved this? | Do you know who approved this? |
| What does this mean? | Could you help me understand what this means? |

## Chapter 4. 수동태 — 디자인/개발 문서에서 자주 보이는 표현

> 영어 문서, 스펙 시트, 이슈 트래커에 수동태가 넘쳐난다. 읽고 쓸 줄 알아야 한다.

### 🔑 수동태가 쓰이는 이유

- 행위자(누가)보다 행위 결과(무엇이)가 중요할 때
- 공식 문서, 스펙, 보고서에서 주어를 흐릴 때
- 책임 소재를 명확히 하지 않을 때

**패턴:** `주어 + be동사 + 과거분사(p.p.)`

---

### 📌 디자인/개발 문서에서 자주 보이는 수동태

| 상황 | 수동태 표현 |
|------|-------------|
| 기능 설명 | This feature **is triggered** by a user action. |
| 상태 설명 | The component **has been updated** in the design system. |
| 이슈 | This bug **was reported** during QA. |
| 승인 프로세스 | The design **needs to be approved** by the PM. |
| 배포 | The new version **will be released** next Thursday. |
| 제한 사항 | Access **is restricted** to admin users only. |
| 변경 사항 | The color tokens **were changed** based on accessibility guidelines. |
| 문서화 | All decisions **should be documented** in Confluence. |

---

### 📌 능동태 vs 수동태 — 언제 뭘 쓸까

| 상황 | 능동태 | 수동태 |
|------|--------|--------|
| 내가 한 일 말할 때 | I updated the mockup. | (수동태 불필요) |
| 누가 했는지 모를 때 | ? | The mockup **was updated**. |
| 공식 문서 | - | Designs **must be reviewed** before handoff. |
| 책임 강조 | The PM rejected the proposal. | The proposal **was rejected** (by the PM). |

---

### 📌 실무에서 자주 쓰는 수동태 문장 모음

**리뷰/승인**
- This design **has been reviewed** and approved.
- Changes **were requested** by the stakeholders.
- The proposal **is being considered** by the team.

**개발 협업**
- The API **has been integrated** into the prototype.
- This component **is expected** to be reusable.
- Specs **should be finalized** before development starts.

**일정/배포**
- The release date **has been pushed** to next week.
- The feature **was shipped** in the last sprint.
- Updates **will be communicated** via Slack.

**이슈**
- The issue **was identified** during user testing.
- A fix **has been deployed** to the staging environment.
- This edge case **hasn't been handled** yet.

---

### ✏️ 변환 연습: 능동태 → 수동태

1. The designer updated the components. → The components **were updated** (by the designer).
2. We will release the new feature next week. → The new feature **will be released** next week.
3. The team has approved the final design. → The final design **has been approved** (by the team).
4. Someone reported a bug in the latest build. → A bug **was reported** in the latest build.
